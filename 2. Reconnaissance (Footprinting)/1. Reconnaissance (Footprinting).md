# 🕵️ Reconnaissance (Footprinting)

**Reconnaissance** is the opening phase of any penetration test or cyberattack — where the attacker gathers as much intelligence as possible about the target before doing anything else. This phase shapes everything that follows; the better the recon, the sharper the attack plan.

## 🔍 Types of Reconnaissance

### 1. Passive Reconnaissance

Gathering information **without ever directly touching** the target.

**Examples:**
- Searching public records
- Combing through social media profiles
- Using tools like WHOIS, DNS lookups, job boards, or company websites

### 2. Active Reconnaissance

Gathering information through **direct interaction** with the target.

**Examples:**
- Scanning ports and services with tools like Nmap
- Sending phishing emails to harvest credentials
- Probing web applications directly for vulnerabilities

---

## 🎯 What Reconnaissance Is Actually For

- Spotting potential entry points
- Understanding the target's network layout
- Collecting details about users, roles, and infrastructure
- Surfacing likely vulnerabilities before exploitation even begins

## 🧩 Key Areas to Focus On

### 1. Network Vulnerabilities
- **Open ports** — unnecessary or unsecured services left exposed
- **Misconfigured devices** — routers, firewalls, switches running weak or default settings
- **Weak protocols** — outdated stuff like Telnet, FTP, or SMBv1 still in use
- **Unauthorized access** — systems that accept connections without proper authentication

### 2. Web Application Vulnerabilities

The classic OWASP Top 10-style issues:
- **Injection attacks** — SQL, NoSQL, command, LDAP
- **Broken authentication** — weak or missing auth mechanisms
- **Cross-Site Scripting (XSS)** — malicious scripts injected into pages
- **Cross-Site Request Forgery (CSRF)** — tricking users into unintended actions
- **Insecure deserialization** — exploiting serialized data to run arbitrary code

### 3. System & Server Vulnerabilities
- **Unpatched software** — missing security updates
- **Default credentials** — factory usernames/passwords never changed
- **Privilege escalation paths** — misconfigurations that let users climb privilege levels
- **Insecure file permissions** — sensitive files reachable by people who shouldn't have access

### 4. Authentication & Authorization Issues
- **Weak password policies** — easily guessable or reused passwords
- **Brute force / dictionary attacks** — straightforward credential-cracking attempts
- **Misconfigured MFA** — weak or simply missing multi-factor authentication

### 5. Data Exposure
- **Sensitive info in code/logs** — credentials, API keys, or PII sitting in source code, error messages, or logs
- **Data leaks** — exposed databases, cloud storage, or backups
- **Weak encryption** — outdated or fundamentally weak encryption schemes in use

### 6. Social Engineering Weaknesses
- **Phishing susceptibility** — employees falling for fake emails or sites
- **Pretexting** — impersonation used to extract sensitive info
- **Physical security gaps** — unsecured physical access to systems or sensitive areas

### 7. Wireless Network Vulnerabilities
- **Insecure Wi-Fi configs** — open or weakly encrypted networks
- **Rogue access points** — unauthorized devices mimicking a legitimate network
- **MITM attacks** — intercepting traffic between devices

### 8. Cloud Security Issues
- **Misconfigured storage** — publicly accessible buckets (e.g. S3)
- **Exposed APIs** — APIs missing authentication or rate-limiting
- **IAM misconfigurations** — overly permissive roles or leaked credentials

### 9. IoT & Embedded System Weaknesses
- **Hardcoded credentials** — fixed login details baked into devices
- **Insecure firmware** — vulnerabilities sitting in the firmware itself
- **Missing updates** — devices that never receive security patches

## 🏢 Organization-Level Information

- **Organization website** — public-facing source of general and technical info
- **Company directory** — employee names, roles, and contact details
- **Employee profiles** — LinkedIn, social media, or other public sources
- **Location details** — physical addresses and office locations
- **Phone numbers/addresses** — valuable for pretexting or phone-based phishing
- **HTML source comments** — leftover developer notes
  *Example: `<!-- TODO: Remove this link before going live -->`*
- **Published security policies** — public statements revealing security posture
- **Linked external services** — third-party APIs or services tied to the main site
- **Company background** — history, structure, ownership
- **News and press releases** — recent expansions, acquisitions, or changes

---

## 🌐 Network-Level Information

- **Domain names** — the org's overall internet footprint, e.g. `example.com`, `sub.example.com`
- **Internal domains** — naming conventions used internally, e.g. `internal.local`, `intranet.example.com`
- **IP addresses** — both public and private ranges
- **Unmonitored/private sites** — internal apps that aren't publicly indexed
- **TCP/UDP services** — what's actually running on which ports, e.g. HTTP (80), SSH (22)
- **VPN/IDS/IPS/access controls** — the network's defensive mechanisms
- **Phone/VoIP systems** — communication infrastructure, useful for social engineering angles

---

## 🖥️ Operating System-Level Information

- **User & group info** — usernames and group memberships
- **Banner grabbing** — pulling software/version info straight from service banners
- **Routing tables** — network routes and interfaces in use
- **SNMP data** — device and network configuration details
- **System architecture** — 32-bit, 64-bit, or other platform specifics
- **Remote systems** — anything connected to or shared with the target
- **System naming conventions** — names that hint at role, e.g. `DC01` (domain controller), `SQLServer`
- **Passwords** — pulled from exposed services, brute-forcing, or social engineering

---

## 🌍 External Reconnaissance

- **Network** — public-facing IPs and domains
- **Phone** — public or leaked phone numbers
- **Website** — analyzing forms, scripts, and overall structure
- **Source code** — public repositories (GitHub, etc.)
- **Website mirroring** — tools like `wget` or `HTTrack`
- **Archive sites** — historical snapshots (e.g. the Wayback Machine)
- **GitHub recon** — tools like Gitrob or truffleHog for leaked secrets
- **WHOIS** — domain registration and contact info
- **Web server content** — exposed backups, configs, or other sensitive files
- **Email headers** — IPs, server paths, and infrastructure hints
- **Search engines** — Google dorking for sensitive or internal data
- **Advanced search operators** — `filetype:`, `site:`, `inurl:`, etc.
- **People-search sites** — services that aggregate personal info
- **Social networks** — LinkedIn, X/Twitter, Facebook, and similar
- **Job postings** — often reveal tech stack details unintentionally
- **Change-monitoring tools** — e.g. Google Alerts, for tracking updates over time

---

## 🏠 Internal Reconnaissance

- **IP addresses** — mapping out the internal network
- **Internal DNS** — domain records and zone transfer attempts
- **Private websites** — intranet or other internal-only applications
- **Dumpster diving** — physically searching discarded materials for sensitive info (printed docs, sticky notes, old hardware)
- **Shoulder surfing** — observing someone's screen or keyboard to capture credentials or data

## 🗺️ Attack Surface Mapping

**Attack surface mapping** is the process of identifying and analyzing every possible point where someone could interact with a target system or organization. It's how you understand the *full* scope of where vulnerabilities could realistically live.

### Key Components

**1. External Attack Surface**
- Public-facing applications (web apps, APIs, VPNs)
- Public IP addresses
- DNS records
- Cloud services
- Third-party integrations

**2. Internal Attack Surface**
- Internal web apps (intranet, admin portals)
- Internal networks and shared drives
- User accounts and privilege levels
- Internal services (databases, file servers)

**3. Physical Attack Surface**
- Office buildings and access points
- Employee devices (laptops, USB drives)
- IoT devices (cameras, sensors)
- Dumpster-diving risk areas

### The Process

1. **Discovery** — identify every relevant asset
2. **Enumeration** — gather detailed info (versions, configs)
3. **Classification** — determine how sensitive each asset is
4. **Risk Assessment** — evaluate vulnerabilities and their potential impact
5. **Visualization** — map it out with diagrams or tables
6. **Prioritization** — focus effort on the most critical assets first

## 🛠️ Tools & Techniques

- **Network scanning** — Nmap, Masscan
- **Web crawling** — OWASP ZAP, Burp Suite
- **OSINT** — Recon-ng, Maltego, Shodan
- **Vulnerability scanning** — Nessus, OpenVAS
- **Cloud-specific tools** — ScoutSuite, CloudMapper
- **Physical recon** — site walkthroughs, social engineering exercises

---

## ✅ Why Attack Surface Mapping Matters

- Gives a clear, honest picture of exactly where the exposure points are
- Enables proactive, rather than reactive, risk management
- Directly informs how security testing and assessments get planned
- Strengthens incident response and ongoing monitoring efforts
